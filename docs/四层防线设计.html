Adaptive Information Retrieval Specification

企业知识库“宏观/微观提问”自适应四层防线重构设计案

本设计白皮书聚焦于破解企业级 RAG 落地中的致命痛点——“检索不足（Under-retrieval）与宏观宏大提问残缺”。 本案彻底抛弃了锁死固定召回数量的传统硬件妥协思路，全链路重构为由 “Query 多意图拓宽、Cross-Encoder 动态分值滑窗、父子块语义联动关联、Agent 反思二次重试” 组成的自适应四层安全隔离防御体系，确保系统在面临任何极端复杂提问时均具备工业级的强鲁棒性与高置信度。

1. 自适应四层防线深度流转拓扑图 (Four-Layer Defense Topology)

注：下流转图严格遵循 Mermaid v10 最新语法标准。当任意层级检测到“知识不充分”时，全自动激活右侧的异步补偿、自动降级与重试机制。

graph TD
    %% 外部提问进入
    UserQuery([用户宏观/微观提问]) --> FA_Gate[FastAPI 网关入口]

    %% 第一层防线：Query 拓宽
    subgraph Layer1[【第一层防线：意图拓宽】]
        FA_Gate -->|调用后台轻量 LLM| QueryRewriter[Query Fusion: 衍生4组近义提问]
        QueryRewriter -->|多线程并行并发查询| QdrantSearch[Qdrant 混合多维向量检索]
    end

    %% 第二层防线：父子块关联
    subgraph Layer2[【第二层防线：父子块级联】]
        QdrantSearch -->|捞回低噪音子块 Child_Chunks| ParentLookup{校验 parent_id 是否存在}
        ParentLookup -->|存在外键关联| SQL_Fetch[SQLAlchemy 级联捞出大文本父块/章节摘要]
        ParentLookup -->|不存在外键| DirectChunk[保持原子细粒度子块]
    end

    %% 第三层防线：动态置信度精排
    subgraph Layer3[【第三层防线：动态阈值精排】]
        SQL_Fetch & DirectChunk -->|合并送入隔离的模型网关| CrossEncoder[Cross-Encoder 交叉编码器算分]
        CrossEncoder -->|设定硬红线 Score >= 0.65| ThresholdCheck{分值动态数量过滤器}
        
        ThresholdCheck -->|大量硬核高分块| FlexWindow[动态打破5个限制: 饱满扩容塞入]
        ThresholdCheck -->|高分块极少/边缘化| RestrictWindow[动态收缩裁剪: 只取高分块去噪]
    end

    %% 第四层防线：Agent 反思审计与自动二次重试机制
    subgraph Layer4[【第四层防线：Agent反思重试】]
        FlexWindow & RestrictWindow -->|拼装硬核上下文提示词| Ollama_Inference[Ollama 核心推理引擎]
        Ollama_Inference -->|前置 Buffer 检测逻辑| AgentAudit{模型反思: 资料是否完备?}
        
        %% 自动降级与补偿回路
        AgentAudit -->|缺失核心参数/判定无法回答| InsufficientFlag([吐出 INSUFFICIENT_INFO 状态标签])
        InsufficientFlag -->|拦截流/触发降级补偿机制| RetryLoop[二次自动重试: 粗筛范围扩大至 Top-50 / 放宽精排]
        RetryLoop -->|重新注入检索网络| QdrantSearch
        
        AgentAudit -->|资料高度完备/自信回答| OutStream[Stream=True 正常流式吐字]
    end

    OutStream -->|前端打字机无感渲染| UserQuery

2. 整合四层防线必须预留的系统级“工程暗桩”（Required Groundwork）

01

第一层预留：FastAPI 并发去重锁与倒数排名融合（RRF）空间

物理原理： 当用户的原始提问被改写拓宽为 4 个衍生句进行多线程并行检索时，Qdrant 向量库在同一毫秒内会返回 4 组完全不同的 Top-20 结果集（共 80 个 Chunk）。这些结果之间必然存在大量的物理重复和排序冲突。

底层预留： 必须在 FastAPI 网关的后台操作模块中预留标准 asyncio.gather 协程并发锁，并且在数据清洗层预留标准的 RRF (Reciprocal Rank Fusion) 倒数排名融合算法函数空间。确保这 80 个块能在内存中以极低的 CPU 开销完成交叉去重、加权重新排序，随后干净地递交给第二层。

02

第二层预留：SQLAlchemy 关系表级联与 Qdrant Payload 的 parent_id 物理外键

物理原理： 解决“微观小块找得准、宏观大块信息全”的物理矛盾。如果只切大块，检索噪音极其严重；如果只切小块，一旦遇到宏观提问，小块拼凑出来的信息就像盲人摸象、残缺不全。

底层预留： 1. **PostgreSQL 端**：在 chunks 表中必须强制预留 parent_id (VARCHAR/UUID, nullable=True) 字段，级联指向存储文章大段落或全章摘要的 parent_documents 表。
2. **Qdrant 端**：在写入的点（Points）的 Payload 字典结构中，必须固定预留 "parent_id" 键。这使得 Cross-Encoder 可以在不感知大文本的情况下对子块精准打分，一旦子块中榜，sync_manager 能够瞬间顺藤摸瓜，用结构化的 Postgres 父块大文本或章节 Summary 动态替换掉子块，完美喂饱大模型。

03

第三层预留：Pydantic 动态阈值 Model 与 Cross-Encoder 绝对值透传

物理原理： “20选5”在传统粗暴的 RAG 里是个死数字。但在高鲁棒系统中，必须转变为由 Cross-Encoder 真实概率得分控制的动态滑窗。

底层预留： 必须在请求和配置的 Pydantic 模型（如 class RerankSettings(BaseModel)）中预留 score_threshold: float = 0.65 和 `max_vram_token_limit` 刚性限制。重排服务返回的绝对置信度分值（0~1）必须原封不动透传给过滤器函数：如果有 15 个块得分大于 0.65，且总 Token 未超出显存红线，系统自动扩容全量带走（解决不够回答的问题）；若只有 1 个块及格，则执行严苛去噪、绝不多带。

04

第四层预留：FastAPI 流式缓冲区拦截与二次重试循环（Retry-Loop）状态机

物理原理： 这是整个系统最后也是最强韧的一道容错底线。大模型在阅读了重排文本后，如果以其原生智能判定“这些资料确实缺乏核心因果关系、数据不足以严谨回答”，则必须触发自适应补偿，而不是强行胡编乱造。

底层预留： FastAPI 在读取 Ollama 的流式响应时，代码层绝不能直接把字吐给前端。必须在异步迭代器中预留一个 **“前置字符缓冲区（Token Buffer Window）”**（约 20 个 Tokens）。在 System Prompt 中锁死：“若资料不足，必须首先吐出 【INSUFFICIENT_INFO】 标签”。一旦缓冲区检测到该魔术字符串，FastAPI 立即实施**流拦截**，在后台将 Qdrant 粗筛的 top_k 从 20 强行开闸放大到 50，放宽重排分值至 0.5，自动触发 **二次检索补偿循环（Retry-Loop）**。只有当二次重试依然判定缺失时，才降级输出友好提示或提示用户追加补充，全流程用户完全无感知。

3. 通用高鲁棒性四层防线核心参数配置白皮书 (Parameter Blueprints)

数据与防线层级 标准工业推荐参数 自适应控损机制与底层物理设计原理
第一层：意图拓宽层
(Query Fusion)		 expanded_queries: 4
rrf_constant: 60
parallel_workers: 4		 将原始问题解构并派生出 4 组涵盖不同侧重点的近义句，利用多线程同时轰炸向量库。采用标准 RRF 倒数排名融合算法去重聚合，大幅提升长文本、宏观提问的“首轮知识召回率”。
第二层：父子级联层
(Parent-Child Schema)		 child_chunk_size: 300 字
parent_chunk_size: 2000 字
join_strategy: lazy_load		 300 字子块负责在 Qdrant 中死守“高相似度、低噪音”的纯净度；一旦被 Cross-Encoder 委以高分，利用 PostgreSQL 的 parent_id 瞬间映射出包含完整技术上下文或全章大纲摘要的 2000 字父块进行内容替换，完美撑大生成层的信息量。
第三层：动态精排层
(Cross-Encoder Fit)		 score_hard_line: >= 0.65
elastic_min_k: 2
elastic_max_k: 12		 彻底废弃死板的固定的 5 个块限制。在 0.65 置信度硬红线之上，若高分干货密集则放宽弹性滑窗至 12 个块，全部打包带走喂饱 LLM；若皆是边缘垃圾，则动态收缩至最极致的 2 个块，严防死守噪音对大模型的干扰。
第四层：反思补偿层
(Agent Retry-Loop)		 buffer_tokens: 20
max_retry_attempts: 2
secondary_top_k: 50		 FastAPI 异步迭代器前置预留 20 个字符缓冲区，专门捕捉大模型的 【INSUFFICIENT_INFO】 异常标签。一旦捕获立即拦截阻断，在后端无缝启动二次扩大检索循环，将 Qdrant 广度瞬间拉开到 Top-50 执行降级补偿，将系统的召回抗风险能力拉到极致。
⚙️
自适应动态弹性滑窗下的显存刚性防爆红线（Dynamic VRAM Protection）：
由于在解决“不够回答”时，第三层防线允许将 Chunk 数量从 5 个动态暴增到 12 个（大文本父块级联），这会导致单用户动态 KV Cache 的显存开销在瞬时呈现翻倍式的阶跃。为了防止极端多用户同时触发宏观提问导致物理显卡瞬间溢出（OOM）挂起，后端 FastAPI 的中间件必须预留并强制绑定以下刚性熔断公式：
当前可用安全动态通道数 = [显卡总物理显存 - 静态模型总计常驻占用] / [单用户动态最大硬核弹性滑窗 Token 占用 (12个父块约 4.8G)] 在系统开发时，必须在 FastAPI 初始化中强制注册一个全局异步计算计数器（Concurrency Global Token Counter），一旦当前线上正在推理的动态总 Token 水位逼近服务器物理上限的 85%，自适应防线必须立即对后续请求启动强制降级硬卡截断（Force Token Clipping），将弹性滑窗强行压回标准 Top-4，牺牲微小的宏观全面性，死守整机硬件生态的绝对永续运行。
