## 一、整体架构图

```
用户提问
  │
  ▼
Dify Chatflow（UI 层，待接入）
  │  HTTP POST
  ▼
FastAPI（api/main.py）
  │
  ├── 权限检查：PostgreSQL → users 表 → RBAC 过滤
  │
  ├── 向量检索：bge-m3 编码 → Qdrant 相似度搜索（Top-20）
  │
  ├── 精排：bge-m3 余弦相似度重排（Top-5）
  │
  └── 生成回答：Ollama qwen3.5:9b

文件放入 /knowledge_base/
  │
  ▼
Watchdog 监听 → 防抖 10s → ingest_document()
  │
  ├── 解析：PyMuPDF / python-docx / openpyxl / python-pptx / ebooklib
  ├── 分块：语义分块（LLM） → 递归分块 → 固定分块（自动降级）
  ├── 元数据：LLM 提取 category / tags / summary
  ├── 编码：bge-m3 via Ollama API
  ├── 存储：Qdrant（向量） + PostgreSQL（元数据）
  └── 归档：复制到 _processed 目录
```

## 二、外部服务（运行在服务器上）

### 1. Ollama — 本地 LLM 推理引擎

- **是什么**：一个本地运行大语言模型的工具，类似"本地版 ChatGPT API"
- **为什么选它**：服务器无法访问 HuggingFace/OpenAI，Ollama 可以离线运行模型，自带 GPU 加速
- **怎么用的**：
  - `qwen3.5:9b` — 中文问答生成（`ollama generate` API）
  - `bge-m3` — 文本向量化（`ollama embeddings` API）
- **部署方式**：系统级服务，端口 11434
- **关键代码**：[bge_embedder.py](pipeline/embedders/bge_embedder.py)、[api/main.py:98-111](api/main.py#L98-L111)
- **调研重点**：Ollama API 文档、`think=False` 参数（qwen3.5 的思考模式会浪费 token）

### 2. Qdrant — 向量数据库

- **是什么**：专门存储和检索向量的数据库。传统数据库用 SQL 查精确匹配，Qdrant 用向量查"语义相似"
- **为什么选它**：开源、轻量、支持 payload 过滤（在向量搜索的同时过滤元数据）、REST API 简单
- **怎么用的**：
  - 入库时：把每个 chunk 的向量 + 元数据（department, project, visibility, status 等）存进去
  - 检索时：先做向量相似度搜索，同时用 payload filter 过滤权限和状态
- **部署方式**：Docker 容器，端口 6333（REST）/ 6334（gRPC）
- **关键代码**：[qdrant_store.py](retriever/qdrant_store.py)
- **调研重点**：Qdrant 的 Filter 机制、Cosine 距离、payload 是什么

### 3. PostgreSQL — 关系型数据库

- **是什么**：传统 SQL 数据库，存结构化数据
- **为什么选它**：存文档生命周期管理数据（谁废弃了什么、审计日志、用户权限），这些用 SQL 查比向量库方便
- **怎么用的**：4 张表
  - `doc_lifecycle` — 文档元数据 + 生命周期状态（主键：doc_id）
  - `users` — 用户信息 + RBAC 角色
  - `doc_audit_log` — 操作审计日志
  - `expire_rules` — 自动过期规则
- **部署方式**：Docker 容器，端口 5433
- **关键代码**：[init_db.py](scripts/init_db.py)
- **调研重点**：SQLAlchemy ORM、表设计、索引

### 4. Dify — LLM 应用开发平台（待接入）

- **是什么**：可视化的 AI 应用搭建平台，可以拖拽创建 Chatflow（对话流程）
- **为什么选它**：不需要写前端代码，通过配置 HTTP 节点调用我们的 RAG API，就能给用户提供聊天界面
- **怎么用的**（计划）：
  - "知识问答" Chatflow：用户输入问题 → HTTP 节点调 `/api/query` → 返回带引用的回答
  - "管理助手" Chatflow：管理员输入指令 → LLM 解析意图 → HTTP 节点调 `/api/admin/*`
- **调研重点**：Dify Chatflow 配置、HTTP 节点、变量传递

## 三、Python 核心框架

### 5. FastAPI — Web API 框架

- **是什么**：Python 的高性能 Web 框架，自动生成 API 文档
- **为什么选它**：异步支持好、自带 Swagger 文档（`/docs`）、Pydantic 数据校验
- **怎么用的**：定义了 `/api/query`、`/api/admin/*`、`/health` 等接口
- **关键代码**：[api/main.py](api/main.py)
- **调研重点**：FastAPI 的 Depends 依赖注入、Pydantic BaseModel、lifespan 生命周期

### 6. SQLAlchemy — Python SQL 工具包

- **是什么**：Python 最流行的数据库 ORM（对象关系映射），用 Python 代码写 SQL
- **为什么选它**：成熟稳定、支持多种数据库、连接池管理
- **怎么用的**：创建引擎 + Session，用 `text()` 执行原生 SQL
- **关键代码**：[api/main.py:42-43](api/main.py#L42-L43)（引擎创建）、[init_db.py](scripts/init_db.py)（建表）
- **调研重点**：Session 生命周期、连接池参数（pool_size, max_overflow）、text() 参数化查询

### 7. Pydantic — 数据校验库

- **是什么**：用 Python 类型注解自动校验数据
- **为什么选它**：FastAPI 内置依赖，定义请求/响应模型自动校验 + 生成文档
- **怎么用的**：定义 QueryRequest、QueryResponse 等模型
- **关键代码**：[api/main.py:116-135](api/main.py#L116-L135)
- **调研重点**：BaseModel、Field 验证、类型注解

## 四、文档解析库

### 8. PyMuPDF（fitz）— PDF 解析

- **是什么**：PDF 文件的读写库，速度快、功能全
- **为什么选它**：比 pdfplumber 快 10 倍+，支持提取文字和内嵌图片
- **怎么用的**：`fitz.open()` 打开 PDF → 逐页 `get_text()` → 提取图片
- **关键代码**：[pdf_reader.py](pipeline/loaders/pdf_reader.py)
- **调研重点**：fitz 的 Page 对象、图片提取（get_images）、CMYK 转 RGB

### 9. python-docx — DOCX 解析

- **是什么**：读取 Word .docx 文件
- **怎么用的**：遍历 paragraphs 提取文字
- **关键代码**：pipeline/loaders/docx_reader.py

### 10. openpyxl — Excel 解析

- **是什么**：读取 .xlsx 文件
- **怎么用的**：遍历 sheet → 遍历 row → 拼接单元格文字
- **关键代码**：pipeline/loaders/excel_reader.py

### 11. python-pptx — PPT 解析

- **是什么**：读取 PowerPoint .pptx 文件
- **怎么用的**：遍历 slides → 遍历 shapes → 提取文字
- **关键代码**：pipeline/loaders/pptx_reader.py

### 12. ebooklib — EPUB 解析

- **是什么**：读取电子书 .epub 格式
- **为什么选它**：EPUB 本质是 HTML 打包，ebooklib 解包后用正则去 HTML 标签
- **怎么用的**：`epub.read_epub()` → `get_items_of_type(ITEM_DOCUMENT)` → 正则去标签
- **关键代码**：[epub_reader.py](pipeline/loaders/epub_reader.py)
- **调研重点**：EPUB 文件结构（ZIP 包含 HTML）、ebooklib API

### 13. chardet — 编码检测

- **是什么**：自动检测文本文件的编码（UTF-8、GBK、GB2312 等）
- **为什么选它**：中文环境很多文件不是 UTF-8，需要自动检测
- **怎么用的**：读取二进制 → `chardet.detect()` → 用检测到的编码 decode
- **关键代码**：pipeline/loaders/text_reader.py

### 14. LoaderRegistry — 加载器注册表（自研）

- **是什么**：用装饰器模式实现的文件格式分发器
- **怎么实现的**：`@LoaderRegistry.register([".pdf"])` 装饰器把 loader 类注册到字典，`get_loader(file_path)` 根据后缀返回对应 loader
- **为什么这么实现**：新增格式只需加一个文件 + 装饰器，不用改分发逻辑（开闭原则）
- **关键代码**：[loader_registry.py](pipeline/loaders/loader_registry.py)

## 五、AI/ML 相关

### 15. bge-m3 — 文本 Embedding 模型

- **是什么**：BAAI（智源）出品的多语言文本向量化模型，把一段文字变成 1024 维的数字向量
- **为什么选它**：中文效果好、支持多语言、1024 维精度够用
- **怎么用的**：通过 Ollama API 调用（`client.embeddings(model="bge-m3:latest", prompt=text)`）
- **为什么不用 HuggingFace 直接加载**：服务器访问不了 huggingface.co，所以用 Ollama 托管
- **关键代码**：[bge_embedder.py](pipeline/embedders/bge_embedder.py)
- **调研重点**：什么是 Embedding、向量维度、余弦相似度

### 16. Reranker — 精排模型

- **是什么**：对初步检索结果重新排序的模型
- **理想方案**：cross-encoder（如 bge-reranker-v2-m3），把 query 和文档拼接输入模型打分，精度高
- **当前方案**：用 bge-m3 的 embedding 做余弦相似度排序（因为服务器下不了 cross-encoder 模型）
- **为什么这么实现**：Ollama 不支持 native reranking API，只能用 embedding 相似度代替
- **精度差异**：cross-encoder ~95% vs embedding 余弦 ~85%
- **关键代码**：[bge_reranker.py](pipeline/rerankers/bge_reranker.py)
- **调研重点**：bi-encoder vs cross-encoder 的区别、为什么 cross-encoder 更准但更慢

### 17. qwen3.5:9b — 大语言模型

- **是什么**：阿里通义千问的 9B 参数模型，用于生成回答
- **为什么选它**：中文效果好、9B 参数在 16GB 显卡上跑得动
- **怎么用的**：`ollama.generate(model="qwen3.5:9b", prompt=..., think=False)`
- **关键注意**：`think=False` 必须加，否则模型会把 token 消耗在"思考"而不是输出回答
- **调研重点**：什么是 temperature、think 模式、prompt engineering

## 六、文件监控

### 18. watchdog — 文件系统监听库

- **是什么**：Python 的跨平台文件系统事件监听库
- **为什么选它**：比轮询高效、支持 Linux inotify / macOS FSEvents / Windows ReadDirectoryChanges
- **怎么用的**：
  - `Observer` 监听 `/knowledge_base/` 目录递归
  - `IngestHandler` 处理 `on_created` / `on_modified` 事件
  - 防抖机制：文件变化后等 10 秒再处理（防止写一半就被读取）
- **关键代码**：[folder_monitor.py](watcher/folder_monitor.py)
- **调研重点**：Observer 模式、inotify 原理、防抖（debounce）概念

## 七、文本处理

### 19. 语义分块（自研）

- **是什么**：把长文档切成适合向量化的小段（chunk）
- **三种策略，自动降级**：
  1. **语义分块**：LLM 分析文档结构，按章节切分（最准，但依赖 LLM）
  2. **递归分块**：按段落 → 句子 → 字符逐级切分（不依赖 LLM）
  3. **固定分块**：按固定字符数切分（兜底）
- **为什么要分块**：Embedding 模型有长度限制（~512 token），太长的文本需要切分
- **为什么要重叠**：chunk 之间保留 overlap，防止切在句子中间丢失语义
- **关键代码**：[splitter.py](pipeline/splitters/splitter.py)
- **调研重点**：chunk_size 对检索质量的影响、overlap 的作用

## 十、设计模式总结

| 模式 | 用在哪 | 为什么 |
|------|--------|--------|
| **注册表模式** | LoaderRegistry | 新增文件格式只需加一个装饰器，不改分发逻辑 |
| **协议（Protocol）** | BaseLoader | 定义 loader 接口，所有 loader 实现统一的 `load()` 方法 |
| **自动降级** | splitter | 语义分块失败 → 递归分块 → 固定分块，保证不卡住 |
| **依赖注入** | FastAPI Depends | `_get_pg()` 自动管理数据库 Session 生命周期 |
| **双写同步** | sync_manager | PG + Qdrant 同时更新，靠 check_consistency 兜底 |
| **防抖（Debounce）** | watchdog handler | 文件变化后等 10 秒，防止写一半就被处理 |
| **装饰器** | LoaderRegistry.register | 注册 loader 到字典，运行时动态分发 |

## 十一、技术选型决策记录

| 决策点 | 选择 | 被否决方案 | 理由 |
|--------|------|------------|------|
| Embedding 方式 | Ollama API | HuggingFace transformers | 服务器访问不了 huggingface.co |
| Reranker 方案 | embedding 余弦相似度 | cross-encoder (bge-reranker-v2-m3) | 服务器下不了模型，Ollama 不支持 reranking API |
| 向量数据库 | Qdrant | Milvus / Weaviate / Chroma | Qdrant 最轻量、Docker 一行启动、payload filter 好用 |
| 关系数据库 | PostgreSQL | MySQL / SQLite | PG 支持数组类型（TEXT[]）、JSONB，适合存 tags |
| Web 框架 | FastAPI | Flask / Django | 异步支持好、自带文档、Pydantic 集成 |
| LLM 托管 | Ollama | vLLM / llama.cpp | Ollama 最简单、自带模型管理、GPU 自动检测 |
| 文件监听 | watchdog | inotify-tools / 轮询 | Python 原生、跨平台、比轮询高效 |
| UI 方案 | Dify Chatflow | 自建前端 | 零前端开发、可视化配置、内置用户管理 |
| 文档同步 | 应用层双写 | CDC (Debezium) | 架构简单、Qdrant 无现成 CDC connector |
