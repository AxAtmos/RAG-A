TECHNICAL REVIEW ONLY

一、 方案设计概述 / Design Abstract

本技术方案旨在解决 RAG（检索增强生成）系统在应对企业多源长文本、宏观提问时由于固定数量召回（如固定 Top-5）所导致的“检索不足（Under-retrieval）”与“长尾语义噪音”问题。系统通过解耦存储架构，部署由 Query 并行拓宽、父子块外键级联、Cross-Encoder 置信度动态滑窗、LLM反思重试状态机 构成的四层防御管道。本设计着重于组件层级接口预留及数据流动流转控制，以保障高并发环境下的整机高HA及数据一致性。

三、 核心数据模型与核心算法接口设计 / Code & Schema Blueprint

为了承载上述四层自适应架构，必须在底层数据表结构（SQLAlchemy）与核心路由处理层预留以下物理字段和逻辑分支。

3.1 关系型存储模型预留 (SQLAlchemy Relational Schema)

设计边界说明： SQL 数据库不存储原始二进制物理文件流（PDF/DOCX等），只保留文件的物理存放路径（file_path）用于归档溯源。利用 Text 字段存放清洗完毕并结构化的长文本父块，提供高并发下的秒级内存检索响应。 

# models.py
from sqlalchemy import Column, String, Text, ForeignKey
from sqlalchemy.orm import relationship
from sqlalchemy.ext.declarative import declarative_base

Base = declarative_base()

class ParentDocument(Base):
    """
    【父表】：存储宏观长文本片段或全章结构化摘要（解耦原始二进制文件流）
    """
    __tablename__ = 'parent_documents'
    
    id = Column(String(36), primary_key=True)
    file_name = Column(String(255), nullable=False)        # 原始文件名追踪
    file_path = Column(String(512), nullable=False)        # 【预留物理路径】：指向本地SSD或MinIO物理路径（禁止存大对象流）
    full_parent_text = Column(Text, nullable=False)        # 2000字左右的纯文本父块，供检索不足时动态替换
    security_level = Column(String(20), default="public")  # RBAC 权限硬连接锁

    children = relationship("ChildChunk", back_populates="parent", cascade="all, delete-orphan")

class ChildChunk(Base):
    """
    【子表】：存储微观细粒度文本片段，对应向量库特征映射
    """
    __tablename__ = 'child_chunks'
    
    id = Column(String(36), primary_key=True)
    parent_id = Column(String(36), ForeignKey('parent_documents.id', ondelete='CASCADE'), nullable=True) # 【核心暗桩】：映射父块外键
    child_text = Column(Text, nullable=False)            # 300字左右的高聚焦子文本
    qdrant_point_id = Column(String(36))                 # 对应 Qdrant 存储的 Point UUID
    
    parent = relationship("ParentDocument", back_populates="children")

3.2 动态滑窗精排与流拦截机制 (Python Async Logic)

# pipeline.py (FastAPI 路由核心处理层伪代码)
async def rerank_and_filter(query: str, raw_chunks: list, settings: RerankSettings):
    """
    第三层防线：基于 Cross-Encoder 置信度绝对分值的动态滑窗机制
    """
    # 构造交叉编码器输入向量对 [ [query, chunk1], [query, chunk2] ... ]
    pairs = [[query, chunk.text] for chunk in raw_chunks]
    scores = await cross_encoder_mesh_service.predict(pairs) # 调用物理隔离的模型网关
    
    valid_chunks = []
    for chunk, score in zip(raw_chunks, scores):
        if score >= settings.score_threshold:  # 预留硬红线阈值控制 (如 0.65)
            chunk.meta['ce_score'] = score
            valid_chunks.append(chunk)
            
    # 根据得分密度动态调整滑窗大小 K 值
    # 预留弹性区间限制：elastic_min_k=2, elastic_max_k=12
    valid_chunks = sorted(valid_chunks, key=lambda x: x.meta['ce_score'], reverse=True)
    return valid_chunks[:settings.elastic_max_k]


async def ollama_stream_interceptor(response_generator):
    """
    第四层防线：FastAPI 异步迭代器 Token 缓冲区前置审计与重试状态机
    """
    token_buffer = ""
    insufficient_flag_detected = False
    
    async for chunk in response_generator:
        # 在大模型吐字的前 20 个 Tokens 内进行缓冲区截留审计
        if not insufficient_flag_detected and len(token_buffer) < 100:
            token_buffer += chunk['response']
            if "【INSUFFICIENT_INFO】" in token_buffer:
                insufficient_flag_detected = True
                # 触发第四层重试降级回路
                new_context = await execute_secondary_retry_loop() 
                async for retry_chunk in call_ollama_with_new_context(new_context):
                    yield retry_chunk
                return
        else:
            # 正常输出流
            yield chunk['response']

四、 生产环境核心组件配置清单 / Production Parameters

配置模块 (Module) 推荐工程参数 (Parameters) 控损机制与物理设计考量
数据分片层 (Slicing) child_size: 300字
parent_size: 2000字
overlap: 45字		 微观子块维持小粒度高语义聚焦，最大化降低向量召回阶段的背景噪音。父块采用按物理章节大纲或固定2K字长度切片，承载宏观业务上下文。
双网关检索层 (Retrieval) vector_top_k: 20 -> 50
ce_threshold: >= 0.65
elastic_k: 2 ~ 12		 Qdrant 基于 HNSW 原生 Payload 级联过滤条件进行第一轮粗筛。Cross-Encoder（交叉编码器）精排模块物理隔离部署。通过分值硬阈值进行弹性自适应过滤，规避固定 K 值导致的信息残缺或噪音倒灌。
大模型推理层 (LLM Engine) num_ctx: 16384 (16K)
temperature: 0.1
think: false (Disabled)		 锁死推理上下文窗口在 16K 水位，控制单物理并发下 KV Cache 显存开销平稳在 ~2.5GB 物理区间。
红线限制： 必须在后端请求体中显式关闭思考链（think=False）。在基于事实支撑的 RAG 业务场景中，长推理步（Reasoning Tokens）不仅增加首字返回延迟（TTFT），且在大并发环境下因吞吐过长会剧烈挤兑系统 I/O 总线带宽。
[VRAM_BOUNDARIES_WARNING]
由于多防线自适应系统允许将检索块动态扩展至最大 12 个长文本父块，系统并发时的单请求 KV Cache 显存开销具有突发阶跃特征。系统初始化阶段必须在 FastAPI 拦截层强制注册动态资源计数器（Semaphore 信号量机制），遵循以下公式实施整卡物理熔断防护：
Max_Allowed_Concurrency = (Total_Physical_VRAM - Model_Static_Weights_Size) / (Max_Elastic_Token_Window_Size [12xParent_Blocks ~4.8GB]) 当线上总激活 Token 空间触及总显存可用容积的 85% 警戒线时，第四层防线自动激活截断补偿（Force Token Clipping），强行把弹性滑窗上限压回 4 个标准块，以牺牲极个别宏观长文本提问的全面性为代价，换取全集群服务的连续可用性。
